快速开始

第一次接管流量，照这个顺序就够了。

  1. 先关闭其他同类客户端（如 ClashX / Verge / Nyanpasu / Party 等），避免系统代理被抢占。
  2. 首次启动时，可先使用默认配置启动一次，以下载 mihomo 所需依赖文件。
  3. 在 Proxy 中选择或导入你的本地 / 远程配置。
  4. 点击 Start 启动 Core；刚改完配置或内核时建议用 Restart。
  5. 在 Proxy Group 中选择节点并执行延迟测试，先确认线路真的可用。
  6. 确认连通性正常后再开启系统代理；如果开关没反应，检查“系统设置 → 通用 → 登录项”里的 ClashBar 后台项目是否已授权。

FAQ

什么是 ClashBar，它适合谁？

ClashBar 是基于 SwiftUI + AppKit 构建、由 mihomo 驱动的原生 macOS 菜单栏代理客户端。它不想做一个臃肿的大壳，而是把配置管理、节点切换、规则刷新、连接排障和系统代理控制收束进菜单栏。

  • 适合的人：想要原生菜单栏体验、轻量体积、长期稳定运行和可观测排障能力的 macOS 用户。
  • 项目状态：这是一个持续与 Codex 协作迭代的自用项目，需求、实现和文档都在快速打磨。

如果你关心内部拆分，可以继续看 ARCHITECTURE.md。

为什么它一直强调“轻量”和“稳定”？

ClashBar 的目标很直接：在保留核心代理能力的前提下，把复杂度、体积和运行负担压下来，不为了短期堆功能去牺牲日常稳定性。

  • 体积目标：打包 mihomo Core 后尽量控制在 20 MB 以内；移除 Core 后尽量控制在 10 MB 以内。
  • 当前对比：ClashBar.app (No Core) 约 7.6 MB，ClashBar.app 约 18.4 MB，ClashMac.app 约 75.2 MB，Clash Verge.app 约 128.4 MB，Clash Party.app 约 496.7 MB。

这些数据来自同一台 macOS 环境下的 Finder 显示值，重点不是炫数字，而是说明这个项目的设计方向。

安装和使用前，哪些前提不能忽略？
  • 只保留一个接管者：同一时间只保留一个 mihomo / Clash 系客户端接管系统代理。别让其他同类软件和它抢端口、系统代理或 Helper。
  • 安装位置要对：优先使用 Release 页面提供的版本，并把 ClashBar.app 放到 /Applications。
  • 登录项要授权：第一次启用系统代理或开机启动时，如果 macOS 弹后台项目提示，就去“系统设置 → 通用 → 登录项”允许 ClashBar。
  • 升级后别先怀疑代码：如果系统代理开关点了没反应，先把登录项里的 ClashBar 后台项目关掉再打开一次。
  • 改完就重启运行态：切换配置、替换内核、远程订阅批量更新后，只要行为异常，先执行一次 Restart。
第一次接管流量，推荐怎么开始？
  1. 点击菜单栏图标，打开 ClashBar 面板。
  2. 在 Proxy 页面选择配置，或者导入本地 / 远程配置。
  3. 点击 Start 启动 Core。
  4. 选择代理模式：Rule、Global 或 Direct。
  5. 在 Proxy Group 中切换节点并执行延迟测试。
  6. 确认连通性正常后再开启系统代理。
哪些场景最适合用 ClashBar？
  • 日常办公：根据网络环境快速切换配置与节点，优先保证在线稳定性。
  • 故障排查：结合 Connections 和 Logs 看连接异常、规则命中和实际路由。
  • 订阅维护：批量更新远程配置后，一键重载并复核节点延迟。
  • 安全优先：敏感信息走 Keychain，日志内容做脱敏处理。
配置、日志和运行状态都存在哪里？

运行时根目录在 ~/Library/Application Support/clashbar。

  • config/：用户配置文件和订阅导入结果。
  • logs/：运行日志和排障输出。排障结束后可以按需清理。
  • state/：应用运行状态和会话数据。建议交给程序维护，不要手动改。
  • core/：运行时 Core 二进制。
如何替换或切换 mihomo 内核？

运行时内核路径是 ~/Library/Application Support/clashbar/core/mihomo。

  • 先执行 Stop，确认当前内核进程已经停掉。
  • 准备目标内核可执行文件，文件名保持为 mihomo。
  • 替换完成后，回到 ClashBar 执行 Start 或 Restart。
macOS 提示“已损坏”或“无法验证开发者”怎么办？

这通常是 Gatekeeper 对未公证应用的默认拦截。

  • 先把应用放到 /Applications/ClashBar.app。
  • 打开“系统设置 → 隐私与安全性”，点击“仍要打开”。
  • 如果还是被拦截，再执行 sudo xattr -r -d com.apple.quarantine /Applications/ClashBar.app 去掉隔离标记。
为什么请求没有按预期走代理？
  • 先确认当前不是 Direct 模式，也没有别的软件覆盖系统代理。
  • 去 Rules 页面看命中规则和策略结果。
  • 去 Connections 定位对应连接，核对目标地址和链路。
  • 最后在 Logs 用关键词过滤交叉验证路由决策。

规则、连接和日志这三处信息拼起来，问题通常就露出来了。
